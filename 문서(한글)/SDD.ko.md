# Lumin 명세 주도 개발

이 한국어 문서가 METHOD-000의 정본이다. [SDD.md](../SDD.md)는 영어 번역본이며, 두 문서가 다르면 이 문서를 따른다.

## 정의

Lumin은 명세 주도 개발(Spec-Driven Development, SDD)을 따른다. 코드를 쓰기 전에 최종 계약과 목적 아키텍처를 정하고, 그 경계 안에서 좁은 프로덕션 수직 슬라이스를 구현한다.

## 워크플로우

1. **라우팅과 문제 정의:** [WORKBOARD.md](../WORKBOARD.md)에서 현재 작업의 owner만 읽는다. 사용자, 문제, 범위, non-goals, 관찰 가능한 완료 조건을 적고 `빠르게`나 `안전하게` 같은 표현은 측정 가능한 기준으로 바꾼다.
2. **설계와 동결:** 제품 명세는 무엇을·왜를, 아키텍처는 어떻게를 소유한다. 경계, 데이터 소유권, 실패 상태, 동시성, 저장, 플랫폼 전달, 검증을 정하고 필요한 검토를 통과한 revision만 동결한다.
3. **수직 슬라이스 구현:** 최종 경계와 타입으로 사용자 동작 하나를 끝까지 구현한다. 새 helper, type, rule, dependency를 만들기 전에 기존 owner를 찾고, 활성 슬라이스에 실제 동작이 없는 crate나 fallback owner를 만들지 않는다.
4. **검증과 종료:** 명세에서 기대값을 역산해 핵심 경로, 현실적인 edge case, hard-stop을 검증한다. 활성 슬라이스 안에서만 중복, 세레모니, 의존성 누수를 정리하고 acceptance를 다시 확인한 뒤 필요한 owner 문서만 갱신한다.

## 최소 명세

큰 변경에는 한 줄 정의, 현재 계약, non-goals, acceptance criteria, verification commands만 요구한다. 작은 변경은 정의, 현재 계약, acceptance만으로 충분하다. 별도 문서는 독립적인 지속 사실을 소유할 때만 만들며 다른 문서의 계약을 복사하지 않는다.

## AI 판단

판단에는 확인한 소스나 증거, 적용 범위, 불확실성을 함께 적고 관찰한 증상과 추론한 원인을 구분한다. 아키텍처를 바꾸는 결정은 owner 문서에 기록하며, 확인하지 못한 clean이나 absence를 승인하지 않는다.
