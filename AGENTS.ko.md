# Lumin v2 저장소 규칙

Lumin v2는 Rust 네이티브 저장소 분석 엔진이자 AI용 영속 쓰기 게이트다. 이 문서는 [AGENTS.md](AGENTS.md)의 한국어 번역본이며, 규칙의 정본은 영어판이다. 두 문서가 다르면 영어판을 따른다.

[WORKBOARD.md](WORKBOARD.md)에서 시작해 현재 작업에 연결된 소유 문서만 읽는다. 계약이나 소유자가 불명확하면 수정 전에 멈추고 불확실성을 밝힌다. 예외는 소유 계약을 먼저 개정하고 필요한 검토를 통과해야 하며, 임의 폴백을 만들지 않는다.

태그: `[Auto]`는 저장소 검사나 CI가 강제하고, `[Review]`는 AI의 근거 있는 판단이 필요하며, `[Both]`는 둘 다 필요하다.

## 용어

- **소유자:** 지속되는 사실이나 정책을 정의할 수 있는 유일한 권한.
- **수직 슬라이스:** 임시 경로 없이 필요한 모든 경계와 수용 검사를 통과하는 하나의 관찰 가능한 동작.
- **투영물:** 저장된 증거에서 파생된 비정본 표현.
- **게이트:** 하나의 gate ID로 식별되는 영속 pre-write/post-write 트랜잭션.

## 열 가지 규칙

1. **단일진실원천 순서를 따른다.** `[Review]`
   - 제품 계약 -> 시스템 청사진 -> 담당 아키텍처 문서 -> 활성 슬라이스 -> 이 파일 순서다.
   - 충돌하면 코드보다 상위 소유 문서를 먼저 개정한다. 하위 계층은 제품을 암묵적으로 재정의할 수 없다.

2. **구현 전에 목적 구조를 동결한다.** `[Both]`
   - 아키텍처 변경에는 설계 검토 1회와 독립 적대적 검토 1회가 필요하다.
   - 검토된 프로덕션 수직 슬라이스만 만든다. 임시 경계가 계약으로 굳으므로 MVP 아키텍처, 수평 스캐폴딩, 빈 미래 크레이트, 임시 소유자, 폴백 엔진을 금지한다.

3. **모든 사실에 물리적 소유자를 하나만 둔다.** `[Both]`
   - Cargo 의존성, 가시성, 프로젝트 소유 타입, 아키텍처 검사로 경계를 강제한다.
   - 서드파티 타입은 소유 크레이트 안에 두고, 도메인 사실은 단계 간 JSON이 아니라 프로세스 내부 타입 모델로 전달한다.

4. **결정론적 실행 모델을 하나만 사용한다.** `[Auto]`
   - [ARCH-001](architecture/001-execution-and-ownership.md)을 따른다. 워커는 불변 입력을 소비하고 소유한 출력을 반환하며, 소유자가 정한 하나의 결정론적 병합 단계만 결과를 합친다.
   - 두 번째 스케줄러나 풀을 추가하지 않는다. `jobs=1`과 `jobs=N`은 같은 의미적 증거를 만들어야 한다.

5. **증거를 정본으로 유지하고 질의 가능하게 만든다.** `[Both]`
   - 증거 저장소가 정본 증거를 소유하며 요약, SARIF, 호환 파일은 투영물이다.
   - 누락, 오래됨, 미지원, 불투명, 실패, 잘림 상태를 그대로 보존한다. 제한 질의는 scope, total, 반환 수, 잘림 여부, continuation을 보고한다.

6. **pre/post를 하나의 영속 트랜잭션으로 다룬다.** `[Auto]`
   - pre-write는 gate ID를 반환하고 post-write는 그 ID를 반드시 요구한다. 에이전트는 intent JSON이나 전송용 파일을 만들지 않는다.
   - 쓰기/쓰기 및 쓰기/의미 읽기 충돌은 fail-closed로 처리하고, 결과 전송 전에는 모든 잠금을 해제한다.

7. **비용이 큰 의존성을 격리한다.** `[Both]`
   - 비용이 큰 의존성은 [ARCH-000](architecture/000-system-blueprint.md)이 지정한 소유 크레이트 안에 둔다.
   - 새 의존성은 측정된 제품 가치와 빌드, 크기, unsafe, 전이 비용 검토가 있어야 한다. 런타임 Cargo, Node 분석 의존성, 소스 폴백은 [PRODUCT-000](specs/000-product-contract.md)과 ARCH-000에 따라 계속 금지한다.

8. **사람이 작성한 정답으로 동작을 증명한다.** `[Both]`
   - 기대 결과는 명세와 사람이 작성한 코퍼스 정답에서 가져오며, 현재 구현이나 레거시 출력에서 복사하지 않는다.
   - 핵심 경로, 현실적인 엣지, 필수 hard-stop을 검증한다. CI 통과를 위해 단언을 약화하거나 임의 cap/timeout을 추가하거나 실패를 삼키거나 검사를 skip하지 않는다.

9. **검증 매트릭스를 따른다.** `[Both]`
   - push 전에는 영향 범위의 로컬 검사만 실행하고, 공유 코어 변경이나 CI 진단일 때만 전체 로컬 매트릭스를 실행한다.
   - 공개 CI가 clean locked build, 전체 코퍼스와 결정론, 패키지, 의존성 정책의 병합 권한을 가진다. CI 실패 시 실패한 주변만 로컬에서 재현한다.

10. **현재 소유 변경만 닫는다.** `[Review]`
    - 수용 조건을 다시 확인하고, 소유 사실이 바뀐 경우에만 소유 명세와 Workboard를 갱신하며, 생성 출력을 제거하고 정확한 검사와 한계를 보고한다.
    - 무관한 정리, 레거시 모듈 통복사, 활성 슬라이스 밖의 사용자 작업 변경을 섞지 않는다.

## 검증 매트릭스

| 변경 | push 전 필수 로컬 검사 | 공개 CI |
| --- | --- | --- |
| 문서만 변경 | 링크, 포맷/공백, 아키텍처 정합성 | clean checkout 문서 검사 |
| 크레이트 하나 또는 코퍼스 사례 | `cargo fmt`, 범위가 좁은 Clippy/테스트, 영향받는 코퍼스와 경계 검사 | 전체 workspace, 코퍼스, 결정론, 의존성 정책 |
| 공유 코어 또는 패키징 | 영향받는 workspace 전체와 패키지 smoke, 전체 로컬 매트릭스는 진단 시에만 | locked Windows/Linux 빌드와 동작 패키지 probe |
